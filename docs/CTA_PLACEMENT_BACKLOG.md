# CTA Placement & Prominence — Improvement Backlog

**Source:** Direct comparison against IndiaMART and IndustryBuying mobile journeys (screenshots reviewed 2026-07-03). Confirmed root finding: both competitor apps decentralize conversion actions to the card level (every product/seller/model card carries its own real CTA) and repeat/vary the primary prompt down the page; our app centralizes to a single sticky CTA per page and renders the secondary action ("Connect via IndiaMART") as unstyled text rather than a button. This is being treated as the top-priority gap — ahead of the smaller items from the earlier comparison (WhatsApp path, GST trust block, response-rate %) — because it affects nearly every screen a buyer touches, not one page.

Nothing in this document has been implemented yet. This is the backlog for review before work starts.

## Engineering stories

### 1. Upgrade ConnectButton visual weight
**Background:** `ConnectButton` (built this session for the masked-calling fix) currently renders its default/collapsed state as plain text, and its "connected" state as a lightly-boxed info panel — neither reads as a primary tappable action the way competitor CTAs do.
**Problem:** Visual weight mismatch undersells a real conversion path.
**Technical requirement:** Restyle both the collapsed (`<button>`) and expanded (`<a>`) states in `src/components/ConnectButton.tsx` to use solid/outline button treatment consistent with the existing `bg-cta`/`border-cta` token usage seen on primary CTAs elsewhere (`ProductDetailView.tsx`'s "Get Quote" button, `BrandMCatView.tsx`'s sticky CTA). Preserve the existing `compact` prop's smaller footprint for dense contexts (Compare cards) — compact should look like a smaller version of the same button, not a different control.
**Acceptance criteria:** Connect button is visually distinguishable as an action (not text) at every size it's used; compact and full variants read as the same design language at different scales; no regression to the masked-number reveal behavior itself.
**Edge cases:** Long brand names in the "connected" state caption shouldn't break the button's fixed height; button must remain legible against both light card backgrounds and the darker hero sections it may appear on (e.g., if reused inside `BrandProfileView`'s banner card in future).

### 2–4. Per-card CTA (Category brand cards / Brand-MCat model cards / Home Featured Models)
**Background:** Three structurally identical gaps — card-wrapping `<Link>`s with no conversion action inside them.
**Problem:** Buyer must scroll back to a single, page-level sticky CTA to act on any card that isn't the one that happens to be nearest the bottom.
**Technical requirement:** For each card type, add a button element inside the existing card markup, `event.stopPropagation()`'d against the card's own navigation handler (the codebase already has two working examples of this exact pattern — the shortlist heart icon and the "View Full Details" link inside `BrandMCatView.tsx`'s model cards — reuse that pattern, don't invent a new one). Wire to `useBuyLeadModal().open(...)` with `buildRfqRequirement()` where a full `Product` object is available (Brand-MCat, Home Featured Models), or the existing brand-level generic requirement text where it isn't (Category's brand cards, which only have `Brand` + `Category`, not a specific product).
**Acceptance criteria:** Tapping the new button opens the RFQ modal correctly pre-filled; tapping anywhere else on the card still navigates as before; no double-firing (modal open + navigation both triggering from one tap).
**Edge cases:** Cards rendered inside horizontally-scrolling rails (Featured Models, brand rails) — confirm the new button doesn't interfere with horizontal swipe/scroll gestures on touch devices.

### 5. Per-card CTA on Compare's supplier cards
**Background:** Compare's only conversion action today is page-level ("Get Quotes From All Sellers").
**Problem:** No way to quote a single seller without first removing the others from the comparison.
**Technical requirement:** In `CompareView.tsx`, add a per-card button calling `openBuyLeadForm` scoped to that one `Supplier`, reusing the same requirement-text logic as `handleCompareQuotes` but with a single-supplier `names` string instead of the joined list.
**Acceptance criteria:** Per-card quote request correctly names only that one seller/brand, not the whole comparison set.

### 6–7. Secondary CTA prompts (Category post-comparison-table / Brand-MCat post-dealer-list)
**Background:** Both core pages currently prompt conversion exactly once, at the very bottom.
**Problem:** No mid-page nudge for a buyer who's finished reading the comparison/dealer content but hasn't scrolled to the bottom.
**Technical requirement:** New inline section (not a second sticky bar — an in-flow prompt card) placed directly after the relevant section in each page's JSX. Must be visually distinct from the sticky footer CTA (different container treatment — e.g., a bordered card rather than a full-width bar) so it reads as a considered second touchpoint, not a rendering bug.
**Acceptance criteria:** Prompt references the current selection where relevant (Brand-MCat: names the selected model); clicking it opens the same modal the sticky CTA would, with the same prefilled content.

### 8. Spike — floating persistent Connect button
**Background:** Structurally different mechanism from everything else in this backlog — not a per-card fix, a potential new global UI element.
**Problem:** Unclear whether this is actually needed once items 1–7 land, or whether it would just add clutter on top of an already-denser CTA surface.
**Technical requirement:** Not scoped for a build yet. Next step is a design review, not an engineering ticket — evaluate against the already-shipped `MobileSearchBar` (top-right) and scroll-aware `BottomNav` (bottom) for visual/positional conflicts before deciding whether to build it at all.
**Acceptance criteria:** N/A — output of this item is a decision (build / don't build), not code.

## Suggested sequencing

1. **Item 1 first, always** — every other task inherits its visual language.
2. **Items 2–3** next (Category + Brand-MCat per-card CTAs) — these are the two core destination pages, highest reach.
3. **Items 4–5** (Home + Compare per-card CTAs) — same pattern, lower-traffic surfaces.
4. **Items 6–7** (secondary prompts) — presentational, can land independently of the per-card work, any time after item 1.
5. **Item 8** — deliberately last, and only as a decision, not a commitment to build.
